# Mouse Laboratory Management System — Prototype

Interactive clickable prototype (mockup) for managing lab mice. **UX-first: built to test the workflow with real lab users before any real backend is written.** Everything runs in the browser with in-memory mock data — no database, no build step, no framework.

UI language is **Thai**. Keep it Thai unless told otherwise.

## Run / verify
- Open `index.html` directly in a browser (or the Launch preview panel). No server needed.
- After editing JS, sanity-check syntax: `node --check js/data.js && node --check js/app.js`
- Data resets on every page refresh (in-memory only) — this is expected.

## Files
- `index.html` — shell; loads Google Font (Sarabun), CSS, then `data.js`, then `app.js`.
- `css/styles.css` — all styles. Uses CSS variables in `:root` (colors, radius, shadow). No preprocessor.
- `js/data.js` — mock data + `Data` helper object. Project 1 (NAFLD) is generated by a loop: 4 shelves × 6 cages × 5 mice.
- `js/app.js` — the whole SPA: `App` controller object (routing, rendering, all dialogs, weighing wizard).

## Architecture
- **`App`** (js/app.js): single global object. `App.init()` runs on DOMContentLoaded. Views are rendered by string templates injected into `#root`. Navigation via `App.go(name, projectId)`; `App.shell(crumbs, body)` wraps the header. Views: `renderLogin`, `renderProjects`, `renderDashboard`, `renderReports`, plus modal builders (`openCagePopup`, `openMouseDetail`, `openTreatForm`, `openDeathForm`, `openHumaneForm`) and the weighing wizard (`startWizard` → `renderWizardStep` → `renderWizardReview` → `wizardSave`).
- **`DB`** (js/data.js): `{ currentUser, projects[] }`. Each project: `{ id, name, shelves, cagesPerShelf, groups[], cages[] }`.
- **`Data`** (js/data.js): derived helpers — `getProject/getCage/getGroup`, `controlGroup`, `inStats(m)`, `latestWeight/prevWeight/weightChange`, `cageAvgWeight/cageAvgChange`, `controlAvgChange`.

## Data model (per mouse)
```
{ id, code, sex('M'|'F'), weights[{date,weight}], remark,
  treatments[],    // Sick Case Report entries: { date, time, vet, signs[], support[], diagnosis, treatment, recommend, note }
  excluded,        // "stopped": kept out of group-average stats, still alive & eats/drinks
  alive,           // false once Death is recorded
  death,           // { type:'natural'|'humane', disposition:'dispose'|'necropsy', note, date }
  careOpen,        // vet case currently open → drives the cage "care" (yellow) colour
  humaneOrder,     // vet order to euthanise: { reason, vet, date }
  necropsy }       // Necropsy Record (only when death.disposition==='necropsy'):
                   //   { date, time, examiner, results:{ [organ]:{v:'N'|'A'|'X', note} }, abnormal, avComment }
```
Cage: `{ id, code, groupId, shelf, position, mice[], water/food:{remaining,added,consumed}, status(legacy), lastRecordDate }`.

## Key domain rules (keep consistent)
- **Auth model — two tiers, purely additive** (see the block comment at the top of `js/data.js`):
  1. **POSITION** (`POSITIONS`, `POSITION_ORDER`) — a user may hold **several**, in `DB.users[].positions[]`: `ADMIN, AV, VET, SCI, ACT, IACUC, QA, AUDIT, EX, OCH, GM, EXTERNAL`. Each has `caps[]` and a `scope`: **`'all'`** = sees every project without being appointed, **`'member'`** = only appointed projects (`EXTERNAL` only).
  2. **PROJECT ROLE** (`ROLES`, `ROLE_ORDER`) — per project, in `project.members[] = { userId, roles[] }`. `PI`/`COPI` are identical to each other; `AHS` does the dosing. `SCI`/`VET`/`ACT` mirror the caps of the system position of the same name **but only inside that project** — that is how an EXTERNAL vet works on project A without seeing project B. For someone who already holds the position it adds nothing, so it reads as a paper appointment; there is **no `nominal` flag** any more.
  **Effective capability = every position's caps ∪ every project role's caps. Nothing is ever subtracted.** Gate through `App.can(cap, project)` / `App.hasAccess(project)` / `App.canEnter(project)` — never test a key directly.
  Key splits: `stop` = **PI/CoPI only**; `treat` = AV/VET; `weigh` = **SCI only** (a PI who wants to weigh must also be appointed Sci of the project); `dosing` = AHS; `cageCare` = ACT; `viewReports` (the graph page) is denied to IACUC/QA/AUDIT/EX/ACT/OCH/GM — it is the researcher's result, not welfare data; `approve` + `manageMembers` + `manageUsers` = **AV** (+ADMIN) — PI lists the requested appointments when submitting and AV maintains them afterwards.
  Helpers: `App.positions` / `positionKeys()` / `positionLabel()` / `hasPositionCap()`, `seesAllProjects`, `myProjectRoles(project)`, `isCreator`, `isApproved`.
- **Death is a two-stage flow**: `reportDeath` (anyone on the team) sets `death.carcass='frozen'` with `disposition:null` — the carcass goes to the freezer; then `handleCarcass` (**SCI/VET only**) sets `carcass:'done'` + `disposition:'dispose'|'necropsy'`, and necropsy opens the 11.3-01 form. `App.deathLabel` renders the frozen state, and `openCagePopup`/`openMouseDetail` show a **จัดการซาก** button only while frozen. The old single `deathStop` capability no longer exists.
- **OCH is view-but-never-enter**: it holds `view` + `ochReport` and **no `enterProject`**, so `App.openProject` routes its card clicks to the safety-report page instead of the dashboard, and `renderDashboard` bounces it.
- **Project roles are dormant until approval**: `myProjectRoles` returns `[]` for a waiting/rejected project unless the user is `project.createdBy` (who acts as PI to fix and resubmit). Every project therefore needs a **`createdBy`** field. AV still sees such projects through its position scope, for review only.
  **Demo switcher**: floating panel bottom-right (`.demo-fab`), grouped into *ตำแหน่งระดับระบบ* (one persona per position) and *บทบาทในโครงการ* (u_pi/u_copi/u_ahs). The latter carry `user.projectRole`, which makes `myProjectRoles` return that role in **every** project so a client can compare views. Real deployment: nobody has `projectRole`; membership drives it.
  Project creation: anyone with an account may request one (`createProject`); the creator becomes **PI** and the project sits at `approval:'waiting'` until AV approves/rejects. Project list & audit log are filtered to accessible projects.
- **Weights are always displayed to 1 decimal place, unit grams.** Use `App.g(v)` (plain) and `App.gs(v)` (signed). Never format grams inline.
- **Cage status colour**: gray = normal, yellow = "care" (any living mouse with `careOpen` or `humaneOrder`). Computed by `App.cageStatus(cage)`.
- **Mouse status colour** (by daily weight change, `App.mouseStatus`): green = gain ≥ `App.GAIN_THRESHOLD` (0.2 g), yellow = 0–0.2 g, red = ≤ 0 (loss/no gain). Dead or stopped mice = no status colour.
- **Group averages** (reports, control comparison) exclude mice where `!Data.inStats(m)` (dead or stopped).
- **Weighing mode** (SCI only): entering it starts `App.weighSession` — all cages go gray and values are cleared ("not weighed yet"); each saved cage turns green. Exit restores real values/colours. The wizard weighs **alive mice only** (`w.mice = cage.mice.filter(m => m.alive)`); dead mice are skipped.
- **Edit mode** (dashboard "✏️ จัดการกรง", `App.editing`): empty slots become "＋ เพิ่มกรง", cages become editable; both open `openCageEditor(p, shelf, pos, cage)` (group + sex + count, or delete). Shelves/cages-per-shelf are editable inline (clamped so no cage is orphaned). All edits are logged via `App.log`.
- **Vet actions**: treatment/close-case/humane-endpoint are only reachable by clicking into an individual **mouse** (not from the cage popup). The per-mouse "⋯" kebab in the cage popup holds, gated by capability: **⚠️ แจ้งผิดปกติ** (`flag`, everyone), **Stop** (`stop`, PI-only), **Death** (`deathStop`).
- **Abnormal flag** (`openFlagForm`, `flag` cap = any member): reports a mouse as "looks abnormal" (not a diagnosis) with a free-text note → sets `mouse.flagOpen`/`mouse.flag = {by,note,date}`, shows an **orange "!"** on the mouse (`App.flagMark`) and turns the cage **orange** (`cageStatus` → `'flag'`, priority below care/danger). Awaits VET: from the mouse detail a VET (`treat`) resolves it via **เปิดเคส (openTreatForm)** / **Humane endpoint (openHumaneForm)** / **✓ ปกติ (clear flag)** — and opening a case, issuing a humane order, or recording death auto-clears the flag.
- **Official lab forms** (ศูนย์สัตว์ทดลอง มช.). PDF originals live in `แบบฟอร์ม/`. **Form option terms that are English on the paper form stay English verbatim — never translate them to Thai** (they are technical terms); Thai is only used for UI chrome (field labels, buttons). Data-entry forms are gated by the `treat` capability (VET):
  - **Sick Case Report** (LA Guide-AF 11.1-02) — the "รายงานอาการป่วย" form (`openTreatForm`). Structured: grouped clinical-sign checkboxes (`App.SICK_SIGNS`), supportive-action checkboxes (`App.SICK_SUPPORT`), diagnosis, treatment, recommendation (`App.SICK_RECO`). Saving pushes a treatment entry and opens the case (`careOpen`). Rendered in mouse detail as sign/support chips + recommendation.
  - **Necropsy Record** (LA Guide-AF 11.3-01) — `openNecropsyForm`, gated by `deathStop` (the gross exam follows whoever recorded the death; the vet later adds the AV Comment), shown only for dead mice whose `death.disposition==='necropsy'`. Per-organ exam by system (`App.NECROPSY_SYS`), each organ = N (Normal) / A (Autolysis) / X (Abnormal, note) + overall abnormal finding + AV comment → saved to `mouse.necropsy`. Read-only view via `App.renderNecropsy`.
  - **Death recording** (`openDeathForm`, per-mouse, `deathStop` cap): captures Date/Time/Reporter + type (natural/humane) + disposition (dispose/necropsy) + note (clinical sign before death) → `mouse.death`. **If disposition is `necropsy`, saving opens `openNecropsyForm` immediately** (you can't choose necropsy without then recording it); cancelling leaves the case with a pending-necropsy button in the mouse detail.
  - **Monitoring Record** (LA Guide-AF 11.1-03) — the day-by-day follow-up ("การรักษา/ติดตามอาการ") derived from `mouse.treatments`.
- **Summary reports** (project-level modals, opened from the dashboard mode-bar, visible to all who can view):
  - **รายงานการตายของสัตว์ทดลอง** (`openDeathReport`) — table of every dead mouse (date, time, cage, ID, group, reporter, type, disposition, necropsy status); rows link to the mouse detail.
  - **บันทึกติดตามอาการสัตว์ป่วย** (`openSickReport`) — per treated animal, the day-by-day treatment timeline (oldest→newest) with a status badge (กำลังรักษา / หายดี / ตายแล้ว); card header links to the mouse detail.
- **Export to PDF** — every form/report has a `🖨️ Export PDF` button. No library: `App.printDocument(filename, bodyHtml)` writes a self-contained A4 HTML doc into a hidden iframe (Sarabun font, `App.PRINT_CSS`) and calls `print()` → user picks "Save as PDF". Doc builders return body HTML (unit-testable in isolation) and **each replicates its CMU paper form** via `cmuHeader` (2-row header band) + bordered `table.form` (`table-layout:fixed` + `COLS4`/`COLS2` colgroups so long checkbox rows wrap within the page):
  - `buildSickCaseDoc` (11.1-02), `buildMonitoringForm` (11.1-03, Day 1–14 daily grid), `buildNecropsyDoc` (11.3-01, 4 ID columns), `buildDeathReportDoc` (11.1-01, Protocol/Action-Plan + No./Date/Time/Cage/ID/Reporter table). `buildSickReportDoc` = one `buildMonitoringForm` per sick animal (page-break between).
  - Per-mouse export buttons (`ฟอร์มป่วย` / `ฟอร์มติดตาม` / `ฟอร์มชันสูตร`) live in the mouse-detail footer; the two project-level exports live in the report modals.
- **Project documents** (`openDocuments`, "📎 เอกสาร" in the dashboard mode-bar): attach important PDFs per project (`project.documents[] = { id, name, size, category, uploadedBy, date, url }`, categories in `App.DOC_CATEGORIES`). View/open is for anyone with access; attach/delete needs `editProject`. **Prototype only** — files live in memory as object URLs (`URL.createObjectURL`), lost on refresh; seed rows have `url:null` (metadata "ตัวอย่าง"). A real backend would upload to object storage + keep signed URLs.
- **Audit log**: `DB.auditLog` is append-only (header "📋 Audit Log"), filtered to projects the user can access (admin sees all). Record every meaningful action with `App.log(action, detail, projectName)`.
- **Top bar** is production-styled: brand + crumbs + Audit Log + (admin) จัดการผู้ใช้ + a **user menu** at top-right (avatar/name → dropdown with "ดูข้อมูลผู้ใช้ & สิทธิ์" and "ออกจากระบบ"). There is no standalone "สิทธิ์" tab.
- **User info & permission page** (`renderRoles`, reached via the user menu → "ดูข้อมูลผู้ใช้") is a read-only reference: capability matrix + the current user's role per project. Member/role assignment is done in-project via `openMembers(p)` (PI/admin only); admins are excluded from the "add member" list (they're superusers). The roles page also has a self-service "🔒 เปลี่ยนรหัสผ่าน" (`openMyPassword`).
- **User management** (`renderUsers`, header "👤 ผู้ใช้", gated by `manageUsers` → ADMIN + AV): CRUD system accounts (`DB.users`, fields firstName/lastName/email/password/**position**; `name` is derived and kept in sync via `makeUser`/on save). The position picker is a 12-button `.pos-grid`; only a full ADMIN may hand out `ADMIN`. Rules enforced: ≥1 admin always (`adminCount`/`isLastAdmin`), an admin can't demote or delete themselves, emails unique, password ≥6 chars. This screen is how an outside researcher gets an **External** account before they can log in and request a project. `App.log` records add/edit/delete/password actions.
- **Create Project** (`renderCreateProject`): centered form; cages/mice are configured in-page via a per-cage wizard (`openCageConfig`) that sets group + sex (♂ blue / ♀ pink) + count. Mouse codes auto-generate as `<cageCode>-<n>` (e.g. A-01-1); `App.freshMouse(code, sex)` builds a mouse with a single starting weight.

## Conventions
- Vanilla JS only, no dependencies. Keep the string-template + delegated-handler style already in `app.js`.
- Match existing Thai microcopy and the compact card/modal styling.
- When adding a mouse field, initialise it in `makeMouse` (data.js) so all generated + seeded mice have it.

## Cross-component invariants — keep these in sync when editing
These pieces are coupled; changing one without the others silently breaks behaviour:
- **Asset cache-bust**: after editing any `js/` or `css/` file, bump `?v=N` in **all four** tags of `index.html` (fonts.css, styles.css, data.js, app.js) — the preview/Pages serve the query-stringed URLs, so a stale `v` shows old code.
- **Font is embedded, not linked**: `css/fonts.css` holds IBM Plex Sans Thai (loopless) as base64 `@font-face` blobs — **never** add a Google Fonts `<link>` back. The family ships **400/500/600/700 only**, so styles.css must not request 800/900 (the browser would fake a synthetic bold). The PDF/print iframe is a separate document and needs the font too: `printDocument` resolves `css/fonts.css` via `new URL('css/fonts.css', document.baseURI)` — keep that when touching the print path, and keep `PRINT_CSS`'s `font-family` in sync with `styles.css`.
- **Capabilities**: never check a role key directly — gate via `App.can(cap, project)`. A new capability must be added to **`POSITIONS[*].caps`** and/or **`ROLES[*].caps`** *and* to **`CAPABILITIES`** (data.js), or it will silently never be granted and won't appear in the two permission matrices on the roles page.
- **Adding/renaming a POSITION**: update `POSITIONS` **and** `POSITION_ORDER` (drives the roles-page matrix, its legend, and the admin user-editor `.pos-grid`), then add a demo persona in `USERS`. Anything printing a position must go through `App.positionKeys()` / `positionLabel()` — never `user.position` (it is an **array**, `user.positions`).
- **Demo personas must not leak position caps**: `u_pi`/`u_copi`/`u_ahs` deliberately hold only `EXTERNAL` so switching to them shows the *project role's* caps and nothing else. Give them `SCI` and they silently gain `weigh`, which hides the rule that a PI cannot weigh. `u_pisci` exists to demo the "PI who is also appointed Sci" case, and `u_vetiacuc` to demo additive multi-position.
- **Position scope vs approval rights are different things**: `seesAllProjects` (`scope==='all'`) decides what the project list shows; `canReview` (`approve` cap) decides who gets review buttons. Don't reuse one for the other — IACUC/QA/EX see everything but approve nothing.
- **Nominal project roles**: `ROLES[k].nominal` (SCI/VET/ACT) must stay capability-free — they only document who is assigned. They render dashed (`.role-chip.nominal`) in `openMembers` and are labelled "(ในนาม)" on the roles page. Never hand them caps; grant via the POSITION instead.
- **Project-role personas**: `user.projectRole` (data.js) drives the `App.myProjectRoles` override — a persona holds that role in every project. New users created in the admin editor have `projectRole: null` (correct — they're real, membership-based).
- **Top-level tabs** (`App.TABS`: โครงการ / งานคลัง / การเงิน): each tab is gated by a capability (`view` / `viewSupply` / `viewFinance`) and rendered by `visibleTabs()` in `shell()`. The bar is **hidden when only one tab is visible**, so most positions see no tabs at all. `tabOfRoute(name)` maps a route to its tab for highlighting — **add new routes there** or they will un-highlight the bar. `homeRoute()` is the landing target after login and every "not permitted" bounce; never hard-code `go('projects')` for that, since **GM has no โครงการ tab** and must land on `supply`.
- **`hasAccess` requires the `view` cap first**, then scope/membership. That is the single thing keeping GM (งานคลัง/การเงิน only) out of every project despite `scope:'all'` — don't "optimise" the `view` check away.
- **Next-phase modules**: `App.MODULES` + `renderModulePlaceholder` back the `supply` / `finance` routes (blank by design, facility-wide — **not** per project). Adding another module means keeping four things in sync: `MODULES`, a `case` in `handleNav`, a line in `go()`, and an entry in `TABS` (+ its capability in `CAPABILITIES`/`POSITIONS`).
- **Non-approved projects aren't enterable**: waiting/rejected projects are "not real yet" — **nobody opens the dashboard** (`renderDashboard` bounces any non-`approved` project). Card clicks route via **`App.openProject(p)`**: PI/admin → **`editProject(p)`** (loads the project into `this.draft` with `editId` and opens the create/edit page), AV/admin → `openProjectInfo` popup, everyone else → toast (blocked). The card ⋯ `cages` action and the rejected-card `edit` action also call `editProject` for non-approved projects. `submitCreateProject` branches on `this.draft.editId`: update-in-place + `approval:'waiting'` (keeps members/documents; rebuilds groups/cages so placeholder mice are regenerated) vs push-new. The "➕ สร้างโครงการ" button clears `this.draft` first so a stale edit-draft never leaks into a new project.
- **Operational gate**: for *approved* projects, every *recording* action (weigh, flag, stop, treat/close-case, humane order, death, necropsy) is gated by **`App.isOperational(p)`** (approved & not `closed`) — so a **closed** (approved) project is view-only. `renderDashboard` (canWeigh), `openCagePopup` (kebab `operational`), and `openMouseDetail` (`canTreat`/`canNecropsy` ANDed with operational) rely on it — add new recording actions the same way.
- **Project state**: `approval` (`waiting|approved|rejected`) is separate from operational `status` (`active|closed`). The project card badge/strip, `openProject` routing, `renderDashboard` guard, `isOperational`, and `openProjectInfo` all branch on these — keep them consistent. New projects: `approval:'waiting'`, `status:'active'`; editing a non-approved project sets `approval:'waiting'` (resubmit).
- **Abnormal flag lifecycle**: `openFlagForm` sets `flagOpen`/`flag`; it is cleared in the saves of `openTreatForm`, `openHumaneForm`, `openDeathForm`, and the mouse-detail "✓ ปกติ" handler. Any new resolution path must clear it. Shown by `App.flagMark` + `cageStatus`→`'flag'` + `.m-badge.flag`.
- **Cage status priority**: `cageStatus` returns danger > care > flag > normal; each needs a matching `.cage.<s>` + `.dot.<s>` CSS rule and a dashboard-legend entry.
- **Project documents** are reachable **only** via `openProjectInfo` → `📎 จัดการเอกสาร` (gated `editProject`). AV approval reviews these files, so never remove that path without providing another.
- **Card click vs buttons**: the project card is opened by a click handler on `.card-open` that ignores `.card-menu`/`[data-act]` targets; every interactive control inside a card calls `e.stopPropagation()`. Add new in-card controls the same way or they'll trigger navigation.

## Out of scope for this prototype (per spec)
KPI dashboard, notifications, audit log, fine-grained permissions, real import/export (Excel export is a client-side CSV), mobile version, DB optimization. Don't build these unless asked.
