# Mouse Laboratory Management System — Prototype

Interactive clickable prototype (mockup) for managing lab mice. **UX-first: built to test the workflow with real lab users before any real backend is written.** Everything runs in the browser with in-memory mock data — no database, no build step, no framework.

UI language is **Thai**. Keep it Thai unless told otherwise.

## Run / verify
- Open `index.html` directly in a browser (or the Launch preview panel). No server needed.
- After editing JS, sanity-check syntax: `node --check js/data.js && node --check js/app.js`
- Data resets on every page refresh (in-memory only) — this is expected.

## Files
- `index.html` — shell; loads Google Font (Sarabun), CSS, then `data.js`, then `app.js`.
- `css/styles.css` — all styles. Uses CSS variables in `:root` (colors, radius, shadow). No preprocessor.
- `js/data.js` — mock data + `Data` helper object. Project 1 (NAFLD) is generated by a loop: 4 shelves × 6 cages × 5 mice.
- `js/app.js` — the whole SPA: `App` controller object (routing, rendering, all dialogs, weighing wizard).

## Architecture
- **`App`** (js/app.js): single global object. `App.init()` runs on DOMContentLoaded. Views are rendered by string templates injected into `#root`. Navigation via `App.go(name, projectId)`; `App.shell(crumbs, body)` wraps the header. Views: `renderLogin`, `renderProjects`, `renderDashboard`, `renderReports`, plus modal builders (`openCagePopup`, `openMouseDetail`, `openTreatForm`, `openDeathForm`, `openHumaneForm`) and the weighing wizard (`startWizard` → `renderWizardStep` → `renderWizardReview` → `wizardSave`).
- **`DB`** (js/data.js): `{ currentUser, projects[] }`. Each project: `{ id, name, shelves, cagesPerShelf, groups[], cages[] }`.
- **`Data`** (js/data.js): derived helpers — `getProject/getCage/getGroup`, `controlGroup`, `inStats(m)`, `latestWeight/prevWeight/weightChange`, `cageAvgWeight/cageAvgChange`, `controlAvgChange`.

## Data model (per mouse)
```
{ id, code, sex('M'|'F'), weights[{date,weight}], remark, treatments[],
  excluded,        // "stopped": kept out of group-average stats, still alive & eats/drinks
  alive,           // false once Death is recorded
  death,           // { type:'natural'|'humane', disposition:'dispose'|'necropsy', note, date }
  careOpen,        // vet case currently open → drives the cage "care" (yellow) colour
  humaneOrder }    // vet order to euthanise: { reason, vet, date }
```
Cage: `{ id, code, groupId, shelf, position, mice[], water/food:{remaining,added,consumed}, status(legacy), lastRecordDate }`.

## Key domain rules (keep consistent)
- **Roles** (`ROLES` in data.js): EC, PI, STOCK, VET, SCI. Only **SCI** has `canWeigh`; only **VET** has `canTreat`. Role is switchable live from the header dropdown (for user testing).
- **Weights are always displayed to 1 decimal place, unit grams.** Use `App.g(v)` (plain) and `App.gs(v)` (signed). Never format grams inline.
- **Cage status colour**: gray = normal, yellow = "care" (any living mouse with `careOpen` or `humaneOrder`). Computed by `App.cageStatus(cage)`.
- **Mouse status colour** (by daily weight change, `App.mouseStatus`): green = gain ≥ `App.GAIN_THRESHOLD` (0.2 g), yellow = 0–0.2 g, red = ≤ 0 (loss/no gain). Dead or stopped mice = no status colour.
- **Group averages** (reports, control comparison) exclude mice where `!Data.inStats(m)` (dead or stopped).
- **Weighing mode** (SCI only): entering it starts `App.weighSession` — all cages go gray and values are cleared ("not weighed yet"); each saved cage turns green. Exit restores real values/colours.
- **Vet actions**: treatment/close-case/humane-endpoint are only reachable by clicking into an individual **mouse** (not from the cage popup). Death & Stop are per-mouse actions in the cage popup table, hidden under a "⋯" kebab menu.
- **Audit log**: `DB.auditLog` is append-only and **visible to everyone** (header "📋 Audit Log"). Record every meaningful action with `App.log(action, detail, projectName)` — keep new state-changing actions logged for transparency.
- **Create Project** (`renderCreateProject`): centered form; cages/mice are configured in-page via a per-cage wizard (`openCageConfig`) that sets group + sex (♂ blue / ♀ pink) + count. Mouse codes auto-generate as `<cageCode>-<n>` (e.g. A-01-1); `App.freshMouse(code, sex)` builds a mouse with a single starting weight.

## Conventions
- Vanilla JS only, no dependencies. Keep the string-template + delegated-handler style already in `app.js`.
- Match existing Thai microcopy and the compact card/modal styling.
- When adding a mouse field, initialise it in `makeMouse` (data.js) so all generated + seeded mice have it.

## Out of scope for this prototype (per spec)
KPI dashboard, notifications, audit log, fine-grained permissions, real import/export (Excel export is a client-side CSV), mobile version, DB optimization. Don't build these unless asked.
